# 日程取消与删除设计

## 目标

同时提供“取消”和“删除”操作，并明确遵循飞书的实际能力：飞书删除日程接口会生成 `cancelled` 墓碑，不支持通过开放 API 物理擦除该记录。

## 行为

- 取消：调用飞书日程 DELETE 接口；日程变为 `cancelled` 后不再出现在列表。
- 删除：调用同一个飞书 DELETE 接口；日程变为 `cancelled` 后不再出现在列表。
- 已有 `cancelled` 日程也不再由列表返回。

## 数据流

页面和 MCP 分别暴露取消与删除操作，两者最终复用现有的飞书删除请求。列表读取在共享的飞书访问层统一过滤所有 `status: cancelled` 日程，因此页面和 MCP 行为一致，无需保存额外状态或引入数据库。

## 界面与工具

- 活跃日程显示“打开 / 编辑 / 取消 / 删除”。
- 已取消日程不再显示。
- MCP 新增 `cancel_calendar_event`；现有 `delete_calendar_event` 保留。两者效果相同，描述明确飞书会将日程置为 `cancelled`。

## 错误处理

- 飞书取消失败时不得从本系统列表消失。
- 重复取消或删除按飞书返回结果处理，并向调用方返回明确错误。

## 测试

- `cancelled` 日程不会由列表返回。
- 取消与删除均调用飞书删除接口。
- MCP 同时暴露取消和删除工具。
